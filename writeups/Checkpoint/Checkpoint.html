00Overview

CheckPoint is a Windows Server 2025 domain controller that strings together four distinct Active Directory primitives, ending on one of 2025's most interesting new attack surfaces: delegated Managed Service Accounts (dMSA) and the BadSuccessor abuse path.

You start with a foothold credential and climb through an AD Recycle Bin restore, a password-reuse pivot, a supply-chain-style extension drop, a dMSA DACL abuse, and finally offline credential recovery from a virtual-machine memory snapshot. Because the box is still active, this page is a conceptual review and attack-path map — not a step-by-step solution.

Active machine

No flags, no instance secrets, and no copy-paste exploitation steps appear on this page. The full technical walkthrough is held privately until CheckPoint retires.

02AD Recycle Bin Abuse

The starting account holds a writable DACL not over a live principal, but over an object in CN=Deleted Objects. When the AD Recycle Bin is enabled, a deleted user is merely tombstoned, and an attacker with write rights over it can restore it back into a normal OU.

The trick that makes it useful: the revived account often answers to a reused password. Reviving the object and spraying a credential you already control turns a dormant DACL into a live, more-privileged identity — here, one with write access to a sensitive file share the original account couldn't touch.

Why it works

Write-over-a-tombstone is easy to overlook in BloodHound because the object isn't in the active set. Always enumerate writable permissions against deleted objects, not just live ones.

03Malicious .vsix Foothold

The share that the revived user can write to is described as an “approved VS Code extensions” drop for a specific engine version. A VS Code extension is just a zipped package with a manifest and JavaScript that runs on activation — so a writable extension share is effectively an execution primitive against whoever (or whatever) installs from it.

Packaging an extension whose activation logic stages and launches a C2 payload, then dropping it on the share, yields a callback as ryan.brooks and the user flag. This is the same class of issue as a poisoned internal package feed: trust placed in a writable distribution point with no integrity check.

04BadSuccessor / dMSA DACL Abuse

This is the heart of the box. Delegated Managed Service Accounts (dMSA) are a Server 2025 feature for migrating a legacy service account to a managed one. The migration is driven by a handful of LDAP attributes — most importantly msDS-ManagedAccountPrecededByLink, which points the new dMSA at the account it “succeeds.”

The BadSuccessor technique (disclosed by Logan Goins / Akamai research) abuses the fact that if you can create or control a dMSA and link it to a target account, the KDC will issue the dMSA a ticket that carries the keys of the superseded account — no password reset, no DCSync. On CheckPoint, ryan.brooks has GenericWrite over svc_deploy and create rights in an OU that holds dMSAs, which is exactly the precondition.

The conceptual flow:

  • Create / weaponize a dMSA whose msDS-ManagedAccountPrecededByLink targets the victim service account.
  • Mark the supersession state so the KDC treats the migration as authoritative.
  • Request a ticket for the dMSA — the response embeds the victim account's key material.
Impact

BadSuccessor converts a single GenericWrite over a service account into full credential takeover of that account, entirely through documented dMSA migration behavior. It needs no elevated rights on the dMSA beyond the ability to set a few attributes.

05Retrospective: What Broke

Problem 1 — Rubeus didn't surface the key I needed

The first pass used Rubeus (tgtdeleg → asktgs /dmsa) to pull the dMSA ticket. It came back with the new dMSA's AES256 session key — but not in the form I needed to authenticate as the superseded account. The RC4 (rc4_hmac) key of the impersonated user simply wasn't exposed in that output, which stalled the pivot to svc_deploy.

Fix: I pivoted back to bloodyAD, authenticating with the Kerberos TGT I'd already obtained for ryan.brooks (the tgtdeleg ticket, used via -k), and ran its add badSuccessor primitive. bloodyAD's output includes the dMSA's previous keys — i.e. the keys of the preceding managed account it supersedes — returning both the RC4 and AES material for svc_deploy. That was the unlock: tool-output parity, not technique, was the blocker.

Lesson

Two tools implementing the “same” attack can expose different fields. When a chain stalls on a missing key/etype, re-run the primitive through a second implementation before assuming the path is wrong. bloodyAD surfaced the superseded-account keys that Rubeus' TGS output didn't hand me directly.

Problem 2 — forensics on a credential, not a flag

The final hop isn't an AD bug at all — it's recovering the local Administrator hash from a VM memory snapshot sitting in a backups share. The conceptual issue and its two practical workarounds are covered in the next section.

06Memory Forensics @ Scale

The backups share holds a virtual machine's memory image (.vmem / .vmsn). Dumping the registry hive's secrets from it yields the local admin hash, which passes straight to a privileged context. The obstacle is purely logistics:

The 2 GB problem

A raw memory image is ~2 GB. Pulling it down over SMB on an HTB VPN is painfully slow, and you can't realistically run Volatility on the target itself.

Two ways through, both used here:

  • Relocate + mount. Copy the memory image from the backups share onto a writable share you control on the same host, then mount that share from your own Kali box using recovered credentials and run Volatility 3 (windows.registry.hashdump) locally. You move the file once, host-to-host, instead of dragging it across the VPN.
  • Parse in place. Skip Volatility entirely and run a snapshot-aware credential tool (vmkatz) directly against the .vmsn on the target, extracting the NTLM hash without ever moving the multi-gigabyte file.

Either path ends with the same local Administrator hash, which is then replayed for the root flag.

07Tooling
  • Enumeration: nmap, NetExec (nxc), bloodyAD, BloodHound
  • Foothold: vsce (extension packaging), Havoc C2
  • dMSA / BadSuccessor: SharpSuccessor, Rubeus, bloodyAD, Get-BadSuccessorOUPermissions
  • Forensics: Volatility 3, vmkatz
08Detection Notes
  • Recycle Bin restores: alert on object reanimation (Event 5137/5139) and on writable DACLs over CN=Deleted Objects.
  • dMSA abuse: monitor writes to msDS-ManagedAccountPrecededByLink and msDS-DelegatedMSAState; treat unexpected dMSA creation as high-signal. This is the BadSuccessor IOC.
  • Writable distribution shares: any share that feeds code into clients (extensions, packages, scripts) needs integrity/signing — WRITE there is RCE.
  • VM memory backups: snapshots are plaintext credential stores; restrict and encrypt them.
